A log line without a correlation ID is a fragment. "User X got 500" tells you nothing if you can't find the requests before it, the auth events, the trace, the breadcrumbs. Correlated logging attaches shared IDs to every log, event, and exception from the same operation, so the trail is searchable as a unit.

## What gets correlated

- **Request ID**, one per HTTP request, generated by middleware, returned in the response, propagated to downstream services
- **Trace ID**, one per distributed trace, OTel-generated, carried in `traceparent` across services
- **User / Session ID**, set after auth, attached to every subsequent log and exception
- **Breadcrumbs** (frontend), the trail of UI events leading up to an error in Sentry

## How in Go (slog)

`logger.With(...)` binds attributes to a derived logger that carries them on every call:

```go
reqLog := logger.With("request_id", reqID, "user_id", userID)
reqLog.Info("fetching cart")
reqLog.Error("cart fetch failed", "err", err)
// both lines carry request_id + user_id
```

`logger.LogAttrs(ctx, ...)` pulls from `context.Context` so handlers don't have to thread the logger through.

## How in Sentry (frontend)

Three layers of scope, applied to every event before it's sent:

- **Global**, every event ever sent (env, release, build info)
- **Isolation**, the current request in Node, the page in browser (the user, the request ID)
- **Current**, local, set via `Sentry.withScope((scope) => { scope.setTag(...); captureException(err) })`. Cleared when the callback exits

Breadcrumbs are Sentry's name for "the trail before this error." The React SDK auto-records UI clicks, console calls, navigation, and XHR/fetch. Add manual breadcrumbs for app events ("added to cart", "opened checkout"). Sentry's UI stacks them above the error so you can replay the lead-up.

## Cross-service

- **Request ID**: backend middleware generates or accepts `X-Request-ID`, sets it on the response, adds it to outbound calls. Downstream service reads the header, uses it as its own request ID
- **Trace ID**: OTel handles it. `otelhttp.NewTransport` on the client injects `traceparent`, `otelhttp.NewHandler` on the server extracts it. The trace ID is the join key across services
